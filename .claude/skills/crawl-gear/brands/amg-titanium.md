# 에이엠지티타늄 (amg-titanium)

- **공식**: https://m.amg-titanium.co.kr (Cafe24 모바일 스킨, 서버렌더 → puppeteer 불필요, plain fetch로 충분)
- 크롤러: `sites/amg-titanium.js` (리스팅+옵션+영문명 사전 파싱). 별도 후처리 2개:
  - `amg-titanium-weight-ocr.py <crawl-json> <out-json>` — 무게(텍스트/OCR 하이브리드)
  - `amg-titanium-color-images.py <crawl-json> <out-json>` — 색상 옵션 상품의 imageUrl을
    실제 색상에 맞게 재배정(아래 10번 항목)
- 마지막 크롤: 2026-07, 83개 (원본 상품 71개 — 아솔로 신발 2개 제외 — 옵션 확장 후 83행).
  무게는 45/71개 상품(63%, 53행) 확보. 나머지는 SET 번들(10개, 의도적 제외)과 스펙 텍스트/이미지가
  아예 없는 상품(케이스류 등 16개)이라 0으로 남음.

## ⚠️ 주의사항
1. **"전체상품"(cate_no=28)이 마스터, 서브카테고리는 부분집합.** 브랜드가 나눠둔 서브카테고리
   (25 티타늄컵/26 수저커트/27 조리도구/42 접시및케이스/46 세트구성)를 다 돌아도 73개 중 55개만
   커버 — 아이젠·가스버너·비너·텐트팩·휘슬·벨트·버클류 29개가 어느 서브카테고리에도 없음.
   그래서 서브카테고리 대신 cate_no=28 하나만 페이지네이션(`&page=N`, `more_total_page` 값만큼)
   순회하고 **상품명 키워드로 33개 카테고리에 재분류**한다 (어댑터 `classifyCategory`).
2. **상품정보제공고시 표에 무게/소재 항목이 아예 없다** (상품명/제조사/원산지/가격만). 대신 무게는
   상품마다 **두 곳 중 하나**에 있다 — `amg-titanium-weight-ocr.py`가 순서대로 시도한다:
   - **텍스트 상품(신형 리스팅, 10개)**: `#prdDetailContent`가 이미지가 아니라 순수 HTML 텍스트고
     "기본정보 ● 중 량 : 16g" 형태로 바로 파싱 가능(OCR 불필요, `html.unescape` 필수 — 안 하면
     `&nbsp;` 등이 안 풀려 정규식이 깨짐).
   - **이미지 상품(마케팅 상세 이미지, 대부분)**: macOS Vision OCR. 두 패턴이 있음: (a) 단일 값
     "재질:.../중량(KG):24g" 처럼 그 상품 전용 스펙 블록(라벨 단위 KG인데 실제 값은 g인 경우도
     있음 — 라벨을 신뢰하지 말고 값 뒤에 붙은 실제 단위 문자로 판단), (b) "제품 : 스푼(구형) /
     무게: 22g", "싱글컵 220 / 무게 42g / 중량 220ml"처럼 **한 이미지에 여러 사이즈·타입을 모은
     비교표** — 여러 상품 페이지가 이 표를 통째로 재사용해서 무게 후보가 여러 개 나온다. 표 안의
     라벨(제품명·사이즈)과 내 상품명·용량(`capacity`)을 매칭해서(용량 일치 +20, 폴딩↔고정형/
     신형↔구형처럼 타입이 다르면 -25) 정확한 행을 골라야 한다 — 컬럼 인덱스가 아니라 라벨로
     매핑하는 원칙(SKILL.md "변형 처리" 참고)을 그대로 적용.
   - **"용량"이 아니라 "중량"이라는 라벨로 ml값이 붙어있는 오탈자성 레이아웃도 있다**(예: "중량
     220ml" — 실제로는 용량을 말함). 라벨 단어로 필터링하지 말고 숫자+단위(ml/g) 패턴 자체로
     찾을 것.
   - **SET/세트 상품(10개)은 의도적으로 제외**한다 — 번들 안 여러 아이템의 무게가 표에 섞여있어
     "이 번들 전체 무게"를 안전하게 결정할 근거가 없다.
   - 그래도 못 찾는 상품(16개, 대부분 케이스류)은 진짜로 스펙 텍스트/이미지 자체가 없는 경우.
3. **색상/사이즈 옵션은 11/73개 상품에만 존재**, 나머지는 단일 상품. 옵션 마크업이 두 종류라
   둘 다 파싱해야 함: `<select option_title="색상">` (표준) 과 `<ul option_style="button"
   option_title="...">` (버튼형, 크램폰 165). **속성 순서가 상품마다 달라**
   `option_title`이 `option_style`보다 앞에 오는 경우가 있음 → 정규식을 속성 순서 무관하게 작성.
4. **옵션 라벨이 색상 하나가 아니라 복합 문자열인 경우가 있다.**
   - 크램폰(165): `"오렌지(230-250mm)"` → 색상+사이즈(mm 범위) 복합, 괄호로 분리.
   - 벨트(162): `"물고기모양그레이버클/레드웨빙/사이즈M"` → `/사이즈X` 접미사가 사이즈, 나머지
     전체가 colorKorean(버클모양+색상 복합 설명). color(영문)는 KO→EN 색상 사전으로 라벨 안의
     색상 단어를 순서대로 찾아 합치는 방식으로 대응(`findColorWords`).
   - 아솔로 신발(163/164, **결과적으로 제외**): `option_title` 자체가 색상(블루/오렌지)이고
     옵션 값들이 신발 사이즈(235~280mm) — 색상이 축 이름에, 사이즈가 값에 들어간 역전 케이스.
5. **"전체상품"에 AMG 자체 상품이 아닌 것도 섞여 있다.** 아솔로(Asolo) 등산화 2개(163/164, 옵션
   확장하면 10행)가 끼워팔기로 같이 노출됨 — 신발은 33개 카테고리에 전용 키도 없고 크롤 대상
   브랜드도 아니므로 `isOtherBrand()`(이름에 "아솔로" 포함)로 아예 제외. 재크롤 시 유사한
   끼워팔기 상품이 새로 생기면 같은 방식으로 필터 추가.
6. **이름만으로 카테고리를 못 정하는 상품 2종(PT-3001/PT-3103, "small"/"large"만 표기)은
   실제로는 진동벨(황동 종)이었음** — 이미지 확인 후 `CATEGORY_OVERRIDE`로 수동 지정.
   "33주년리미티드에디션"(144)도 이름에 단서가 없어 이미지로 확인(볼/컵/수저/국자/집게
   혼합 대형 세트) 후 cookware_etc로 수동 지정.
7. **"케이스"로 끝나는 단독 케이스 상품과 "OO(...케이스포함/케이스증정)" 표기를 구분**해야
   한다 — 괄호를 제거한 뒤 "케이스"로 끝나면 그 자체가 케이스 상품(pouch), 아니면 다른 상품에
   케이스가 딸려온다는 뜻(원래 카테고리 유지). 순서를 지키지 않으면 "시에라컵 케이스" 같은
   케이스 전용 상품이 컵(cup)으로 잘못 분류된다.
8. **capacity는 이름에서 첫 매치만 뽑는다** — "싱글컵 SET(220/340/450ml/케이스포함)"처럼 여러
   용량이 한 상품(세트)에 뭉쳐 있으면 스키마상 숫자 하나만 저장 가능해 대표값 하나(정규식이
   먼저 찾는 값)만 남는다. 완벽하지 않지만 세트 상품 특성상 감내.
9. **nameKorean 맨 앞의 브랜드명("에이엠지티타늄"/"AMG티타늄"/"AMG TITANIUM")은 제거한다**
   (`stripBrandPrefix`) — company/companyKorean에 이미 브랜드가 있어 중복. 단 **끝에 붙는
   "...캠핑 백패킹 AMG TITANIUM"류 접미사는 그대로 둔다**(사용자가 "앞단"만 지정). name(영문)은
   stripBrandPrefix 이후의 nameKorean을 음역하므로 자동으로 같이 반영된다.
10. **색상 옵션 상품의 사진이 전부 대표이미지 한 장으로 똑같이 들어간다(사용자가 발견) — Cafe24
    옵션(select/버튼)의 `link_image` 속성이 이 사이트는 항상 빈 값**이라 옵션별 이미지를 마크업
    에서 바로 못 읽는다. 그런데 상세페이지 갤러리(`#xans-product-mobileimage` 안,
    `style="display:none"`으로 숨겨진 `<li>`들)에는 색상별 사진이 실제로 들어있다 — 다만
    **라벨이 없고, 이 상품이 안 파는 색(예: 휘슬S는 블루/그린/그레이/네이비만 팔지만 갤러리엔
    레드 사진도 섞여 있음 — 사이즈 라인업 전체가 공유하는 자산으로 추정) 이 섞여 있어 순서로
    매핑하면 틀린다.** `amg-titanium-color-images.py`가 각 갤러리 이미지의 대표색(HSV, 흰
    배경 제외 후 hue/채도/명도)을 계산해 그 상품이 실제로 파는 색상들과 최적 매칭(그리디, 최대
    거리 120 넘으면 매칭 포기하고 대표이미지 유지)한다. 무채색(회색/검정/네이비/실버)은 hue가
    불안정해 채도<0.15 조건으로 먼저 걸러 명도로 구분. 밴트(162)처럼 색상명이 "물고기모양
    그레이버클/레드웨빙" 같은 복합 문자열(로마자 음역 폴백)인 경우엔 매칭 안 되고 원래 대표
    이미지로 남는다 — 알려진 한계.
