# 코오롱스포츠 (kolon-sport)

- **사이트**: https://www.kolonsport.com (Next.js + Apollo GraphQL 자사몰. `/graphql_private`
  엔드포인트가 인증·영속쿼리 해시 없이도 완전한 쿼리 텍스트를 그냥 POST로 받아준다(introspection만
  막혀있음) — puppeteer 불필요, fetch만으로 크롤)
- 크롤러: `sites/kolon-sport.js`
- 마지막 크롤: 2026-07, 845개 상품/3771행. 의류·신발·가방·용품·등산장비·캠핑 6개 대분류,
  22개 리프 카테고리(기획전/컬렉션 페이지는 실제 카테고리와 상품이 중복되어 제외).
  무게 39/845(4.6%) — 텐트(100%)·침낭(100%)·텐트부속(100%)·매트(100%)·스토브(100%)·
  쿡웨어(100%)·백팩(79.2%) 등 실제 스펙이 있는 카테고리는 이미지 안 스펙표까지 전수 확인
  (§14) 후 확보했고, 의류는 108개 무작위 표본 전수 확인 결과 진짜 제품 무게 언급 0건
  (§15 — 전부 "모델 정보: 키172cm 몸무게51kg" 보일러플레이트 또는 "무게를 최소화한" 류
  정성적 마케팅 문구, 숫자 없음)이라 낮은 게 정상. 규칙 준수 검증 완료(빈 필수값·
  color/size 페어·groupId 형식·중복·HTML엔티티·specs 스키마 키 전부 0건).

## ⚠️ 주의사항

1. **리스팅·상세 전부 GraphQL 하나로 해결된다.** 리스팅: `products(where: {categoryId, sort:
   "newProduct-desc", page, pageSize}) { results { code } page { totalCount } }` — 색상 단위로
   한 항목씩 나온다(사이즈는 안 나옴). 상세: `productDetail(where: {productCode, isNaverEp:
   false}) { product { ... } }` — 이름·설명·소재(`fabric`)·색상(`style`/`swatchColors`)·성별
   (`productGender`)·사이즈 변형(`variantOptions[].legacySize`)·베이스코드(`baseProductCode`)
   전부 여기서 나온다. GraphQL 서버가 이상한 필드 요청 시 "Did you mean X { ... }?" 같은 친절한
   검증 에러를 주므로 스키마 문서 없이도 필드를 하나씩 추측해 발견 가능.
2. **구조화 `product.weight` 필드는 사이트 전체에서 사실상 항상 0이다** — 텐트·백팩·신발·장갑·
   스틱 등 카테고리 가리지 않고 실측 전부 0. 실제 무게값은 `description`(HTML) 마케팅 문구
   안에 자연어로 박혀 있다: "무게 : 14.84kg" / "무게 930g (레인커버 제외)" / "1,350g의
   초경량 무게" / "중량 : 본체 : 149g" / "총 무게 – 총 무게 : 25.3kg" 등 어순·중첩 라벨이
   제각각이라 정규식 여러 형태를 동시에 허용해야 한다(라벨→값 정방향 + 값→라벨 역방향 +
   라벨과 값 사이에 부품명 한 겹 허용).
3. **"사이즈 및 무게 가이드"라는 섹션 제목이 있어도 무게 자체가 없는 경우가 흔하다**(실측:
   하이킹화 다수 확인 — 총길이/전체높이/굽높이 같은 치수만 있고 무게 수치는 단 하나도 없음).
   섹션 제목만 보고 "이 상품은 무게가 있다"고 오판하지 말 것 — 실제 숫자+단위가 있는지로
   판별해야 한다.
4. **"무게" 바로 앞 명사가 본품이 아니라 동봉된 부속품의 무게인 경우가 있다**(실측: 트레일
   러닝 베스트 설명의 "*소프트 플라스크 무게 : 34g" — 베스트 자체 무게가 아니라 같이 오는
   수통의 무게이고, 이 상품엔 다른 무게 언급이 아예 없음). 알려진 부속품 명사(플라스크·
   풋프린트·그라운드시트·스터프색·이너텐트·파우치·케이스·커버)가 바로 앞에 있으면 그 매치는
   버리고 0으로 둔다.
5. **다른 브랜드 세션(thegearshop 등)의 무게 플라우저빌리티 상한(20kg)을 그대로 가져오면
   안 된다** — 코오롱스포츠는 5~6인용 대형 쉘터를 정상 판매하며 실측 "50주년 기념 에디션
   빅돔"이 25.3kg으로 20kg을 넘는 정상값이다. 50kg으로 넉넉히 잡았다.
6. **`swatchColors`는 색상 "계열"만 주는 대략값이라 서로 다른 색상이 뭉개진다** — 실측:
   "카키 그레이"와 "차콜 그레이" 둘 다 `swatchColors: ["GREY"]`. 이걸 그대로 color(영문)로
   쓰면 groupId+color+size 매칭에서 두 색상이 같은 문서로 합쳐져 하나가 유실된다(실측
   50건). 정밀한 `style`(한글, 예: "카키 그레이")을 로마자 음역해 color로 쓰고, `style`
   원문을 colorKorean으로 쓴다.
7. **사이즈는 카테고리 리스팅에 없고 상세조회의 `variantOptions[].legacySize`에만 있다** —
   리스팅은 색상 단위 항목만 주므로(사이즈 축은 상세페이지 진입 후에만 노출), 색상마다
   별도로 상세조회를 해야 그 색상의 사이즈 목록이 나온다. "XXX"는 사이트 자체의 "사이즈
   없음(One Size)" 표기.
8. **`fabric`(소재) 필드에 값이 없다는 걸 나타내는 자리표시자가 다양하다** — "상세설명 참조"
   (주로 텐트·기어류), 단순 "-", 라벨만 있고 콜론 뒤가 빈 문자열("겉감:"), `null` 전부
   확인됨. 이 넷을 전부 빈 문자열로 처리해야 한다.
9. **"매트/침낭"(105010130500), "캠핑 주방"(105010130400) 두 카테고리는 사이트 자체
   분류가 우리 33개 스키마보다 거칠어서 상품명 키워드로 세분류가 필요하다.** 매트/침낭:
   "침낭"→sleeping_bag, "패드/매트"→mat, "풋프린트/그라운드시트/이너"→tent_acc(텐트 부속,
   본체 아님). 캠핑주방: "수저/커트러리"→cutlery, "스토브/버너"→stove, "컵"→cup,
   "보울/그릇"→bowl, 나머지(쿠커/주전자/쿡웨어세트)→cookware_etc.
10. **국내 전용 브랜드라 영문 상품명이 없다** — `name`(영문)은 로마자 음역(RR 테이블,
    Backcountry/CLAYMORE/TheGearShop 세션에서 확립한 것 재사용)으로 채운다.
11. **가끔 "중량(g) 68 72 76"처럼 사이즈별 무게를 표로 나열하는 상품이 있다**(실측: 트레일
    러닝 힙벨트 — 허리둘레 3단계별 무게를 헤더+숫자나열로 표기, 라벨-값이 한 줄에 안 붙어
    있음). 이런 표 형태는 현재 정규식으로 못 잡는다 — 발견 빈도가 낮아(vest_pack 8개 중
    1개) 별도 파서를 만들지 않고 0으로 남겨둠. 추후 vest_pack 카테고리 무게 확인 시 참고.
12. **일부 상품은 `description`이 완전히 비어있다**(CSS 보일러플레이트만 있거나 길이 1) —
    최근 등록된 신상품이 상세페이지 콘텐츠를 아직 안 채운 경우로 추정, 소스 자체 결측치.
13. **`specs` 필드를 카테고리 스키마 확인 없이 넣으면 스키마 밖 키가 생긴다** — `gender`는
    `specs-schema.js`에서 backpack/vest_pack(BACKPACK_LIKE)에만 있는데 전 카테고리에
    무차별로 넣어 3734행이 스키마 위반(clothing.gender 2970행 등)했었고, `material`은
    tent/sleeping_bag엔 아예 없는데(innerMaterial/flyMaterial 등 별도 필드를 씀) 4행에
    잘못 채워졌었다. **스펙 필드를 추가할 때마다 반드시 `SPECS_SCHEMA[category]`에 그 키가
    실제로 있는지 확인할 것** — 카테고리별로 넣고 끝내지 말고 크롤 후 전체 행에 대해
    `Object.keys(specs)`가 스키마 키 집합의 부분집합인지 검증하는 걸 표준 체크리스트
    항목으로 삼는다.
14. **⚠️ `description`이 사실상 비어있어도(길이 1, `<img>` 태그 하나뿐 등) 정말 무게가
    없다고 단정하면 안 된다 — 그 이미지 자체가 스펙표일 수 있다** (실측, 사용자가 실제
    사이트 스크린샷으로 지적: 텐트 "에어로라이트 4"의 description은 `/medias/
    KEXBX24921KHA-4.jpg?context=...` 이미지 하나뿐인데 그 안에 에어로라이트2/3/4 비교표
    "미니멈 무게 1.54/2.14/2.37kg, 풀 패킹 무게 1.73/2.33/2.55kg"가 있었고, 침낭
    "AEROLITE SLEEPING BAG 500/300"도 `images.kolonmall.com/upload/default/2026/*.jpg`
    이미지 하나에 "Fill Weight/Carry Bag/Mesh Bag/Total Weight"(500→768g, 300→554g)
    표가 있었다. **판별 기준: 이미지 URL이 그 상품 코드 전용으로 생성된 것인지 확인** —
    `/medias/<productCode>-N.jpg`나 `upload/default/`처럼 상품별 고유 파일이면 스펙 이미지
    후보이니 반드시 열어봐야 하고, `ks-product-detail/function/water_repellent.jpg`나
    `item_schema/KM_35.gif`처럼 **여러 상품이 똑같은 파일명을 공유**하면 범용 아이콘/치수
    다이어그램일 뿐이라 스펙이 없다(실측: 무게 없는 백팩 5개는 전부 이 공유 아이콘만
    있었고, 상품 전용 이미지가 아예 없었음 — 이 5개는 진짜 결측 확정). 텍스트로 자동
    추출이 안 되니 Claude vision으로 직접 열어 확인한 값만
    `OCR_VERIFIED_WEIGHTS_G`(product.code 기준)에 수동 등록한다.
15. **의류 무게 커버리지가 낮다고 바로 "정상 결측"으로 단정하면 안 된다 — 반드시 큰 표본으로
    재검증한다.** 사용자가 "무게 커버리지가 너무 낮다"고 지적해서 재조사한 결과:
    - `product.weight`(구조화 필드), `extendedProductDetail`, `sizeInformationsForSet1/2`,
      `sizeSpecsForSet1/2`, `productDiagramForSet1/2` 전부 의류·신발·기어 가리지 않고 항상
      `null`/`0`이다(사이트가 안 씀).
    - `product.sizeSpecs`(치수표)는 총길이/가슴둘레/어깨너비 등 **치수만** 있고 무게 행이
      단 한 번도 없다(아우터 40개 표본 확인 — "에그라이트"(초경량 마케팅 라인)조차 예외
      없음).
    - `description` 텍스트에 "무게"/"중량" **단어**는 의류 상품 37%에서 나오지만(108개
      표본 중 24개 이상), 실제로 열어보면 전부 가짜 신호다: (a) 거의 모든 상품 페이지에
      박혀있는 "모델 착장 정보: 키 172cm 몸무게 51kg"(모델 신체 정보, 상품과 무관 —
      "몸무게"의 "무게"가 매치되거나 괄호 없이 "무게 : 51kg"로도 나와 필터링 까다로움),
      (b) "무게를 최소화한"/"가벼운 무게" 같은 정성적 마케팅 문구(숫자 없음). **108개
      표본 전수 확인 결과 진짜 제품 무게 숫자는 0건.**
    - 반면 신발의 "사이즈 및 무게 가이드" 섹션은 제목만 그렇고 실제로 치수만 있는 경우가
      흔하다(§3) — 이것도 처음엔 "그림 안에 무게가 있나?" 의심해 다이어그램 이미지를
      OCR까지 해봤지만(순수 치수 안내 일러스트, 무게 텍스트 없음) 결국 진짜 결측으로 확인.
    - **결론: 낮은 커버리지 자체가 버그의 증거는 아니다 — 하지만 "낮다"는 피드백을 받으면
      (a) 사이트의 다른 GraphQL 필드(`extendedProductDetail` 등 안 써본 필드) 전부 확인,
      (b) 텍스트 패턴을 큰 무작위 표본(384개가 아니라 최소 100개 이상)으로 재검증, (c) 표
      형태("중량(g) 68 72 76")처럼 기존 정규식이 놓치는 다른 서식이 있는지 확인 — 이
      세 가지를 다 해본 뒤에만 "진짜 결측"이라고 결론 내려야 한다.**
