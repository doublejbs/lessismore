# 미스테리월 (mystery-wall)

- **사이트**: https://www.mysterywall.com (Cafe24 자사몰, 국내 전용 브랜드). puppeteer

  상세이미지도 초기 HTML에 바로 있음, lazy 스크롤 불필요).
- 크롤러: `sites/mystery-wall.js`, 무게 OCR: `mystery-wall-weight-ocr.py`
- 마지막 크롤: 2026-07-22, 194개 상품/309행 (사이즈-드롭다운 변형 버그 수정 후 287→309행
  증가, 아래 §0 참고). D-PACK·SHELTER·BACKPACK·POUCH·PACKING·ACC.·AS/PART 카테고리
  (기획전/Fabrics/SALE/BEST 탭은 실제 카테고리와 상품이 중복되어 제외).
  무게 237/309행, 164/194개 상품(84.5%) — 텍스트 크롤 단계에서는 0%(전부 이미지 안에
  있음), 별도 OCR 패스로 확보. 규칙 준수 검증 완료(빈 필수값·color/size 페어·groupId
  형식·중복·HTML엔티티·한글/영문 누출 전부 0건). Firestore push/커밋 아직 안 함(사용자
  명시적 요청 대기).

    대응 colon_value 티어) 155/194 → OCR 3차(결측 34개 전수 육안 재검 후 welght오타·잡음기호

    1,316/1,596/1,806은 오름차순으로 P5/P7/P8. **P5-알파=본체 1,316g(폴 410g), P7-알파=1,596g
    (폴 544g), P8-알파=1,806g(폴 686g)**. (첫 스크린샷 때 "P5=1,596"으로 알려준 건 실제 P7이었음 —
    비교표는 사람이 직접 봐야 모델↔값 매핑이 확실하다.) 티피쉘-라이트(76)=스킨 4,500g(센터폴 810g
    별도, 이미지의 "Weight:1450g"은 증정 그라운드시트 무게), 카이트원 타프(66)=630g.

## ⚠️ 0. 가장 중요한 버그 이력 — 사이즈가 "별도 select 축"으로만 존재하는 상품 (사용자 발견)

초기 어댑터는 색상 아닌 select는 전부 무시했는데, 이 사이트는 사이즈를 두 가지 **서로 다른**
방식으로 표현한다 (§1 참고): 별도 URL(D-PACK류)뿐 아니라 **같은 페이지 안의
`<select title="사이즈" name="option1">` 드롭다운**으로도 존재한다. 이걸 놓치면 무게뿐 아니라
가격이 다른 SKU 자체를 통째로 잃는다.

- **사용자가 스크린샷으로 직접 지적**: "큐브파우치(다이니마) - s 18g, m 20g 로 이미지에 무게가
  있어" — 조사 결과 사이즈 드롭다운을 가진 상품이 총 7개(275, 276, 258, 269, 111 — 사이즈 축 /
  149, 237 — 폴세트·선택옵션류 커스텀 축)나 있었고, 전부 사이즈별 행이 통째로 누락되어 있었다.
- **함정**: 진짜 상품 옵션(`name="option1"`/`option2`)과 Cafe24 "함께 구매하면 좋은 상품"
  교차판매 위젯(`name="addproduct_option_name_NNN"`, 다른 상품의 productNo를 가리킴)을
  `option_title` 속성만으로 구분하려 하면 이 위젯에 오탐된다 (CLAYMORE 세션에서 이미 확인된
  함정, 여기서도 재확인). **반드시 `name="option\d"` 정확 매치**로 select를 스캔한 뒤, title로
  색상/비색상(사이즈 등) 축을 분류해야 한다.
- **수정**: `extractDetail`에서 `name="option\d"`인 모든 select를 스캔 → title에 "색상/컬러"가
  있으면 color 축, 아니면 size/variant 축으로 분류 → 가격 접미사(`stripPriceSuffix`, 예:
  "M (+₩1,000)" → "M") 제거 → 두 축을 cross-product해서 행 전개. 재크롤 결과 287→309행,
  상품 수는 194 그대로(전개만 늘어남).
- **일반 교훈**: 스크린샷 같은 직접 증거로 "이거 틀렸다"는 지적을 받으면, 지적된 사례만 패치하지
  말고 같은 근본 원인이 다른 곳에도 있는지 전수 조사할 것 — 여기서도 2건 지적이 7건 버그로 확장.

## ⚠️ 주의사항

### 사이트 구조 / 변형 처리
1. **변형(색상/사이즈)이 두 가지 메커니즘으로 나뉜다 — 헷갈리면 안 된다.**
   - **색상**은 상세페이지의 `<select title="색상" name="option1">` 드롭다운으로 처리된다
     (같은 productNo, 여러 색상이 옵션값으로만 존재). 실측: "스마트케이스 (X-Pac)"이
     화이트/블랙/딥블루/코요테브라운 4색 옵션.
   - **사이즈**는 완전히 독립된 상품 URL/productNo로 나뉜다(같은 상품명에 "(XS)"/"(S)"/
     "(M)"/"(L)" 접미사만 다름, 드롭다운 전혀 없음). 실측: "디팩 3.0 (XS)"=20, "(S)"=19,
     "(M)"=18, "(L)"=16 — 전부 다른 productNo. `groupId`는 이 접미사를 뗀 베이스 이름으로
     묶어야 사이즈별로 다른 상품 취급되는 사고를 막는다.
2. **`groupId` 슬러그화 시 "+"를 그냥 버리면 안 된다** — 실측: "사코슈 페더+ (다이니마)"와
   "사코슈 페더 (다이니마)"가 둘 다 "+"를 버리는 슬러그 규칙 때문에 완전히 같은 슬러그로
   충돌해 서로 다른 두 상품이 하나의 groupId로 뭉개짐(품절/재입고 표시가 아니라 진짜
   "+" 상위 등급 표시). "+"는 버리기 전에 "-plus-"로 치환해서 구분을 보존해야 한다.
3. **D-PACK 계열은 배낭이 아니라 파우치다** — 상세 스펙 이미지에 "eXtra small (20~30L
   배낭용)"이라고 명시되어 있어, 배낭에 넣어 쓰는 내부 정리용 파우치임을 확인. `backpack`이
   아니라 `pouch`로 분류.
4. **"ACC."(47)/"AS-PART"(48) 카테고리는 게이터·양말부터 텐트 스테이크·폴까지 잡다하게
   섞여 있다** — 상품명 키워드로 세분류 필요(게이터→gaiter, 양말→clothing, 텐트/쉘터/
   타프/펙/폴/스토퍼/와펜/스트링→tent_acc, 나머지→etc).
5. **색상 옵션 텍스트에 재고 상태나 병기된 영문이 섞여 나온다** — 실측: "레드 [품절]"/
   "블랙 [일시품절]"(재고 상태 표시, 색상명이 아님 — 떼어내야 함), "차콜 Charcoal"/
   "화이트 White"(한글+영문 중복 병기 — 영문 부분만 채택). `stripColorStatusSuffix`/
   `colorToEnglish`로 처리.
6. **색상이 영어 색상명의 한글 음역인 경우 그냥 로마자 표기하면 이중 변환 사고가 난다**
   (실측: "화이트"를 로마자화하면 "Hwaiteu" — 원래 "White"였던 걸 다시 로마자로 바꾸는
   왜곡). 자주 나오는 색상(화이트/블랙/네이비/베이지/브라운/샌드/올리브/인디고/차콜/
   옐로우/레드/블루/딥블루/멀베리/라군/캐롯/코요테브라운)은 `COLOR_KO_EN` 사전으로 직접
   매핑, 사전에 없는 순수 한글 색상명만 로마자 폴백.

### 무게/소재 — 전부 이미지 안에 있다 (가장 중요)
7. **무게/소재는 텍스트가 아니라 상세페이지의 `/web/upload/NNEditor/...` 이미지 안에
   그림으로 박혀 있다**(코베아 케이스와 동일 패턴) — 어댑터 크롤 단계의 텍스트 파싱으로는
   무게 완비율이 0%다. 어댑터는 이 이미지 URL들을 `_specImages` 배열에 모아두기만 하고,
   `mystery-wall-weight-ocr.py`가 별도 패스로 Claude vision(macOS Vision Framework) OCR을
   돌려 채운다(CLAYMORE 세션과 동일한 2단계 워크플로 — `node crawl.js` 먼저, 그 다음
   `python3 mystery-wall-weight-ocr.py <json>`).
8. **스펙 이미지 레이아웃이 상품 라인마다 다르다** — (a) "Size & Weight" 다크 패널
   (SIZE/WEIGHT/COLOR 3줄, D-PACK 계열), (b) "Weight : 1450g" 단순 텍스트 한 줄(쉘터
   그라운드시트 계열), (c) "중량 : 470g"처럼 한글 라벨+콜론+값(백팩 계열). 라벨 자체가
   OCR에서 깨지거나(예: "사이즈"→"40|X") 아예 인식이 안 되고 콜론만 남는 경우도 있다
   (실측: "X20 (X-Pac)" — "중량"이 통째로 사라지고 ": 470g"만 남음, 폰트 스타일 문제로
   추정) — 라벨→값(정방향) / 라벨 없이 ":숫자단위"만 있는 경우(콜론 전용 폴백) 등
   4단계 폴백으로 대응.
9. **라벨과 값 사이에 부가 설명이 끼는 경우가 있다**(실측: 게이터 "Weight Per Pair : 88g"
   — "Per Pair"가 라벨과 콜론 사이에 껴서 괄호 허용만으로는 못 잡음) — 라벨 뒤 최대 20자
   까지 느슨하게 허용해서 값까지 도달하도록 정규식을 넓혔다.
10. **이 브랜드 최중량 라인업(대형 백팩)도 3.5kg 안쪽이라 플라우저빌리티 상한을
    5000g으로 낮게 잡았다**(CLAYMORE의 20000g 상한을 그대로 가져오면 OCR 오인식을
    거르는 효과가 약해짐 — 실측 "라이노 95L+" 3400g이 최중량, 5000g 상한이면 이 값은
    통과하면서도 오인식 방지 효과는 유지).
11. **의류(캐시미어 울양말) 1개 상품은 스펙 이미지에 무게 자체가 없다** — 사이즈/소재
    정보만 있고 무게 언급이 전혀 없음, 진짜 결측치.
12. **macOS Vision이 특정 폰트에서 라벨 텍스트를 통째로 인식 못 하는 경우가 있다** (사용자가
    스크린샷으로 지적) — "미니 카고"(productNo 110)는 이미지 y≈3480 지점(6142px 이미지, 타일
    경계 0/1500/3000/4500/6000과 무관한 중간 지점)에 "무게 : 530g | 사이즈 : ... | 소재 : Nylon
    Super-Tex"가 육안으로 선명하게 보이는데 Vision이 해당 영역에서 텍스트를 아예 인식 못 함(글자
    깨짐이 아니라 완전 무반응). 정규식/파이프라인 버그가 아니라 Vision 자체의 폰트 렌더링 인식
    한계로 판단, `MANUAL_OVERRIDES_G = {"110": 530}` 같은 수동 확정값 딕셔너리로 우회(다른
    브랜드의 `OCR_VERIFIED_WEIGHTS_G` 패턴과 동일). OCR이 "못 찾음"으로 나온다고 바로 결측
    처리하지 말고, 스펙 이미지를 직접 열어 육안 확인하는 절차가 필요함을 보여준 사례.
12-b. **한글 무게 라벨("무게"/"중량")이 이 사이트 폰트에서 OCR로 통째로 깨지는 게 매우 흔하다**
    (사용자가 backpack/pouch/shelter/tarp 재OCR 요청하며 발견) — 값(숫자+g/kg)은 잘 인식되는데
    라벨만 알아볼 수 없게 뭉개진다(실측: "무게 : 214g"→"97l : 214g", "무게 : 188g"→"97| : 188g",
    "무게 : 510g"→"Hl : 510g", "무게 : 258g"→"7l : 258g", "무게 : 80g"→"971 : 80g" 등). 라벨
    기반 매칭(WEIGHT_RE·row_paired)이 전부 실패하므로, **최후 티어 `COLON_VALUE_RE`로 "콜론 뒤에
    오는 무게값"을 라벨 무관하게 잡는다**(`[:：][^g:：\n]{0,8}?(숫자)(g|kg|...)`, 콜론 뒤 최대
    8자 잡음 허용 — "PX|: 9 1.35 Kg"처럼 파편 낀 경우 대응). 오탐 방지: (a) g/m²(원단 밀도)는
    음의 전방탐색으로 제외, (b) 데니어("1680Denia")·치수("cm")는 단위가 g가 아니라 자연히 배제,
    (c) **상품당 서로 다른 g값이 정확히 하나일 때만 채택**(여러 개면 모호 → 결측). 이 티어 하나로
    +18개 상품 무게 확보(트윌백 182g, 클램힙색 214g, 팝업팩 188g, D25 510g/D32 592g, 보틀쿨러
    40/50g, 헥사곤 그라운드시트 570g 등, 전수 컨텍스트 검증해 전부 진짜 무게줄임을 확인).
12-c. **비교표 이미지는 자동 추출이 위험하다** — 비트쉘 P5/P7/P8-알파(189/190/191)는 상세
    이미지가 3모델 비교표라 본체값이 여러 개(1,316/1,596/1,806g) 잡힌다 → colon_value 단일값
    가드에 걸려 자동 채움 안 됨(정상). 사용자 확인값(P5=본체 1,596g)만 수동 등록, P7/P8은 모델↔값
    매핑을 신뢰할 수 없어 미채움으로 남김. 향후 이런 비교표는 개별 확인 필요.
12-d. **쉘터 무게는 본체+폴세트 총중량으로 기록한다**(사용자 확정, 2026-07-22) — 스펙이 "무게 :
    본체 1,596g, 폴세트 544g"처럼 분리 표기하면 **둘을 더한 총 캐리 무게**를 넣는다(폴 포함 제품
    한정). 단, (a) 이미지에 폴 무게가 없으면 있는 값만 넣는다 — 실측: 비트쉘 P5/P7-베타는 "Weight :
    1570g (skin only)"처럼 스킨만 명시되고 폴 무게가 이미지에 없어 스킨 무게(1,570/2,060g)만 기록.
    (b) 폴이 필요 없는 제품은 본체=총중량 — 실측: 헥사곤 알파(폴 불필요, 1,350g), 카이트원/알파인
    사각 실타프, 그라운드시트류. 확정 총중량: P5-알파 1,726(1316+410), P7-알파 2,140(1596+544),
    P8-알파 2,492(1806+686), 티피쉘-라이트 5,310(스킨 4500+센터폴 810). ⚠ 티피쉘 5.3kg 때문에
    `MAX_PLAUSIBLE_G`를 8000으로 올렸다.
12-e. **수동 오버라이드는 스칼라/딕셔너리 두 형태를 지원한다** — `MANUAL_OVERRIDES_G` 값이
    스칼라면 상품 전체(색상만 다른 행에 균일 적용), **딕셔너리면 사이즈별 확정값**(그 사이즈 행에만
    적용, dict에 없는 사이즈는 0 유지). 실측: 패커블 파우치(269)는 S/M/L/XL 드롭다운(가격 증가=더
    큼)인데 이미지엔 "Small size 58g"만 있어 `{"S": 58}`로 S만 채우고 M/L/XL은 미채움. 미니파우치
    (93)는 사이즈 드롭다운 없는 단일 SKU에 "1SET (2 size)" S 38g+M 62g 세트라 합산 100g.
12-f. **결측 보고 전 반드시 남은 상품의 raw OCR을 육안 재검한다** — "스크립트가 못 찾음 = 진짜
    무게 없음"이 아니다(사용자 지적으로 발견). 결측 34개 raw OCR을 다시 뜯어보니 이미지엔 무게가
    멀쩡히 있는데 아래 사유로 막힌 케이스가 있었다:
    - "weight"가 "welght"로 깨짐(i↔l 혼동, 실측: 폰 포켓 "• Welght. 56g") → WEIGHT_RE/LABEL_ONLY_RE
      철자에 `we[il1]ght` 퍼지 허용.
    - 값에 잡음 기호가 붙음(실측: 사코슈 페더+ "92g/") → STANDALONE_TOKEN_RE를 `^\W*값\W*$`로
      앞뒤 비단어문자 허용.
    - 사이즈별 단독 토큰이 라벨 없이 나열됨(실측: 이너폼 S/M/L "72g"/"78g"/"84g") → standalone
      포지셔널 티어 추가(값 개수=행 수 且 사이즈로 행이 나뉜 경우만, `positional_ok` 게이트).
    - 노이즈 g값이 섞여 2값 충돌(실측: 헥사곤 알파 본체 "1.35 Kg" + 폴 스펙 "16.5cm 9H : 12g")
      → 수동 오버라이드. 3종세트 합산(페더파우치 10+14+18=42g)도 수동.
    - "무게처럼 보이지만 아닌 g값" 주의: 가스파우치의 "230g/450g"은 담는 가스캔 용량이지 파우치
      무게가 아니다 — 자동/수동 모두 채우지 말 것.
13. **사이즈가 다른 행에 단일 OCR 무게값을 균일 적용하면 안 된다** — 포지셔널 매칭
    (`ocr_label_positional`, 이미지 안 순서=사이즈 옵션 순서로 가정, 큐브파우치 S=18g/M=20g에서
    검증됨)을 추가한 뒤, 이미지에 무게값이 **하나만** 검출된 상품에 대해 그 값을 상품의 모든 행에
    무조건 적용하는 로직이 있었는데, "패커블 파우치"(269, S/M/L/XL 4행)가 이미지 안의 "Small
    size ... WEIGHT 58g"(Small 전용 값)을 M/L/XL에도 그대로 우겨넣는 사고가 발생(다른 사이즈가
    더 무거울 게 거의 확실한데도). **1차 수정**(`n_rows==1`일 때만 균일 적용)은 269는 고쳤지만
    "미니 카고"(110, 색상만 다른 2행)를 회귀시킴 — 행 개수만으로는 색상 때문에 여러 행인지
    사이즈 때문에 여러 행인지 구분이 안 됨. **최종 수정**: 행 개수가 아니라
    `distinct_sizes = {r.get("size","") for r in by_product[pno]}`로 판단 — 사이즈 종류가
    1개 이하(=색상만 다름)일 때만 균일 적용, 사이즈가 실제로 여러 종류면 단일값을 적용하지 않고
    0으로 남겨 결측 처리. 색상-축 vs 사이즈-축 구분이 필요한 모든 향후 브랜드에 재사용 가능한
    일반 원칙.
